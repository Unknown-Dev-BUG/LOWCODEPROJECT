# AI Rules & Project Guidelines (`AI_RULES.md`)

ยินดีต้อนรับ AI Agent ทุกตัวที่เข้ามารับช่วงพัฒนาต่อในโปรเจกต์นี้ โปรดอ่านและปฏิบัติตามกฎและแนวทางการทำงานอย่างเคร่งครัด เพื่อรักษาความต่อเนื่องและความเสถียรของโค้ดเบส

---

## 🚀 1. Technology Stack & Architecture

โปรเจกต์นี้คือ **LowcodeProject** ซึ่งใช้โครงสร้างแบบ Monorepo (Refine):

- **Core Framework**: React + Refine
- **UI Integration**: [Ant Design (`@refinedev/antd`)](file:///packages/antd) (ใช้สำหรับการพัฒนาหน้าบ้าน คอนเทนเนอร์ และ CRUD components ทั้งหมด)
- **Routing**: [React Router v6 (`@refinedev/react-router`)](file:///packages/react-router)
- **Backend Database & Auth**: **Local Supabase (Docker)**
  - **API URL**: `http://localhost:54300` (หรือพอร์ตที่เครื่อง Local รันอยู่)
  - **Studio URL**: `http://localhost:54321` (เข้าสำหรับจัดการ Schema ตารางข้อมูล)
- **AI Engine**: **Gemini AI API** (Google Generative AI)
  - วัตถุประสงค์: ใช้สำหรับการอ่านและถอดความข้อมูลจากไฟล์เอกสาร (Document Data Extraction)

## ⚙️ 2. กฎทองสำหรับ AI Agent (AI GOLDEN RULES)

### 📌 กฎข้อที่ 1: การพัฒนาต้องทำในเครื่องแบบ Local

- ฐานข้อมูล Supabase จะต้องรันผ่าน Docker ในเครื่องเสมอ ห้ามต่อฐานข้อมูลทดสอบนอกเครือข่าย ยกเว้นการเรียกใช้งาน **Gemini API** ที่ต้องขอสิทธิ์ออกอินเทอร์เน็ตเฉพาะการส่งข้อมูลประมวลผลเท่านั้น

### 📌 กฎข้อที่ 2: ใช้คอมโพเนนต์สำเร็จรูปของ Refine + Ant Design เสมอ

- หลีกเลี่ยงการสร้าง UI หรือสไตล์ขึ้นมาใหม่ตั้งแต่ศูนย์ ให้ใช้คอมโพเนนต์จาก `@refinedev/antd` เป็นหลัก เช่น:
  - `<List>`, `<Create>`, `<Edit>`, `<Show>` สำหรับหน้าเพจ CRUD
  - `<Table>` ของ Ant Design ร่วมกับ `useTable` ของ Refine สำหรับแสดงตารางข้อมูล
  - `<Form>` ของ Ant Design ร่วมกับ `useForm` สำหรับกรอกข้อมูล

### 📌 กฎข้อที่ 3: ใช้ Inferencer ในการสร้างโค้ดเริ่มต้น

- เมื่อต้องการสร้างหน้าจอ CRUD ใหม่ ให้ใช้ระบบ **Auto Code Generation** จาก `@refinedev/inferencer` เสมอ เพื่อจำลองหน้าจอและดูรูปแบบก่อน จากนั้นจึงคัดลอกโค้ดมาปรับปรุงตามความต้องการของ User

### 📌 กฎข้อที่ 4: การจัดการโครงสร้างข้อมูลแบบปลอดภัย (Data Safety)

- ข้อมูลไฟล์และเอกสารของ User ให้เก็บไว้ที่ **Supabase Storage (Local)** เท่านั้น ห้ามส่งไฟล์ทั้งหมดไปเก็บข้างนอก
- ในการใช้งาน Gemini ให้ส่งเฉพาะเนื้อหาไฟล์หรือข้อมูลดิบผ่าน REST API เฉพาะจุดที่ต้องประมวลผลเท่านั้น

โปรดศึกษาและปฏิบัติตามคู่มือระดับมืออาชีพอย่างเคร่งครัด:

- 🤖 **[คู่มือพัฒนาฟังก์ชันร่วมกับ Gemini API](file:///ai_memory/gemini-api-dev.md)** (การเรียกโมเดลล่าสุด, โครงสร้าง SDK ใหม่)
- 🗄️ **[คู่มือการออกแบบฐานข้อมูลที่ดี](file:///ai_memory/database-design.md)** (การทำ Schema, ดัชนี, RLS ใน Supabase)
- 🌐 **[คู่มือออกแบบช่องทางสื่อสาร (API Patterns)](file:///ai_memory/api-patterns.md)** (โครงสร้าง API, การเขียน Endpoint ที่ปลอดภัย)
- 📦 **[คู่มือและประวัติแพ็กเกจย่อยในระบบ (PACKAGES.md)](file:///ai_memory/PACKAGES.md)** (สเปกแพ็กเกจหลัก, หน้าที่การทำงาน, วันสร้าง/แก้ไขล่าสุด)
- ⚙️ **[คู่มือการจัดการเวอร์ชันของระบบ (VERSIONING.md)](file:///ai_memory/VERSIONING.md)** (ขั้นตอนและคำสั่งอัปเดตเวอร์ชันโครงการ)

เพื่อป้องกันการเขียนโค้ดที่ไม่ปลอดภัยหรือโค้ดที่ไม่มีคุณภาพ AI ทุกตัวต้องปฏิบัติตามมาตรฐานนี้:

### 📌 กฎเหล็ก TypeScript: ห้ามใช้ `any` เด็ดขาด

- **ห้ามเขียน `any`**: การระบุชนิดตัวแปรเป็น `any` จะทำให้สูญเสียระบบความปลอดภัยของประเภทข้อมูล หากไม่ทราบชนิดตัวแปรที่แน่ชัด ให้ใช้ `unknown` หรือกำหนด Generic Type แทน หรือเขียน Interface/Type ประกาศไว้ล่วงหน้าอย่างละเอียด

### 📌 มาตรฐานความปลอดภัยตามกรอบ OWASP Top 10

1. **ป้องกัน Injection (การฝังคำสั่งอันตราย)**:
   - ห้ามเขียน String ต่อคำสั่ง SQL ดิบเพื่อสั่งรันข้อมูล
   - ให้ใช้คำสั่งมาตรฐานผ่านตัวช่วยของ **Supabase JS Client** เสมอ เพราะมีการทำ Parameterized Query อยู่ภายในตัวโดยอัตโนมัติ
2. **ป้องกันข้อมูลลับรั่วไหล (Cryptographic Failures)**:
   - **ห้ามฮาร์ดโค้ด** API Key ของ Gemini หรือ Token ส่วนตัวใด ๆ ลงในไฟล์โค้ดเด็ดขาด
   - ให้ใช้ตัวแปรสภาพแวดล้อม (Environment Variables) ผ่านไฟล์ `.env` เช่น `import.meta.env.VITE_GEMINI_API_KEY` เสมอ
3. **ป้องกันช่องโหว่สิทธิ์การเข้าถึงข้อมูล (Broken Access Control)**:
   - ในการแสดงผลหรือการทำรายการที่ต้องการความปลอดภัย ให้ตรวจสอบสิทธิ์การเข้าถึงผ่านระบบ **Access Control** ของ Refine (เช่นใช้คอมโพเนนต์ `<CanAccess>` หรือฮุก `useCan`) ก่อนประมวลผลเสมอ
4. **ป้องกันช่องโหว่ XSS (Cross-Site Scripting)**:
   - หลีกเลี่ยงการใช้คำสั่งเรนเดอร์ HTML ตรง ๆ อย่าง `dangerouslySetInnerHTML`
   - หากมีฟังก์ชันถอดความข้อความที่มีความจำเป็นต้องใช้ HTML ให้รันผ่านตัวทำความสะอาดโค้ดอย่าง **DOMPurify** ก่อนแสดงผลเสมอ

## 🌿 7. นโยบายการจัดการกิ่ง Git (Git Branching Strategy)

AI Agent ทุกตัวต้องปฏิบัติตามมาตรฐานการจัดการกิ่งใน Git เพื่อการพัฒนาอย่างเป็นระบบและปลอดภัย:

### 📌 กฎเหล็ก: ห้ามบันทึกงานตรงเข้าสู่กิ่ง `main` เด็ดขาด
* **ห้าม commit หรือ push** เข้าสู่กิ่ง `main` โดยตรงในระหว่างกระบวนการพัฒนาทั่วไป
* กิ่ง `main` จะถูกสงวนไว้สำหรับซอร์สโค้ดรุ่นปล่อยใช้งานจริง (Production-ready) ที่ได้รับการตรวจสอบและยืนยันผ่านกระบวนการ Pull Request (PR) แล้วเท่านั้น

### 📌 ขั้นตอนการทำงานที่ต้องปฏิบัติตาม (AI Development Git Workflow):
1. **สร้างกิ่งแยกสำหรับการพัฒนา**: เมื่อได้รับคำสั่งให้เขียนฟังก์ชันหรือแก้ไขบั๊ก ให้ทำการแตกกิ่งย่อยออกจากกิ่งพัฒนาหลักโดยใช้รูปแบบชื่อดังนี้:
   * **`feature/[ชื่อฟีเจอร์]`** สำหรับการเขียนฟังก์ชันใหม่ (เช่น `feature/gemini-ai-extract`)
   * **`bugfix/[ปัญหาที่พบ]`** สำหรับการแก้ไขบั๊กทั่วไป (เช่น `bugfix/fix-upload-crash`)
2. **Commit งานในกิ่งแยก**: ทำการเขียนโค้ดและ commit ผลงานในกิ่งย่อยของคุณ
3. **ส่งขึ้นระบบและขอความเห็นชอบ (Pull Request)**: สั่ง `git push` กิ่งแยกนั้นขึ้นสู่ GitHub และรายงานผลแจ้งให้ผู้ใช้ทราบว่าพร้อมสำหรับการตรวจสอบ (Pull Request) เพื่อให้ระบบ CI ทำงานและผู้ใช้พิจารณาตรวจสอบความเรียบร้อยก่อนรวมโค้ด (Merge) เสมอ
